3. Arduino IDE Setup & Complete Guide

3.1 Chapter 1: Arduino Development Environment Installation and Configuration

1.1 Arduino IDE Download and Installation

Goal: Install Arduino IDE 2.x (recommended) or 1.8.x on Windows / Mac / Linux.

Steps:

Visit Arduino official website:

https://www.arduino.cc/en/software

Download the corresponding installation package according to your operating system:

  • Windows: Download .exe (recommended: "Windows Win 10 and newer")
  • macOS: Download .app (select Intel or Apple Silicon based on your chip)
  • Linux: Download AppImage or use snap install arduino

Run the installer and follow the prompts to complete the installation (on Windows, it is recommended to check the option to install USB drivers).

Launch Arduino IDE.

💡 Tip: If the official website is slow to access, you can use domestic mirrors (such as arduino.me) or offline packages provided by board manufacturers.

1.2 Add ESP32-S3 Official Board Support

YD-ESP32-S3 development board uses ESP32-S3 chip, which requires adding ESP32 board index in Arduino IDE.

Steps:

  • Open Arduino IDE, go to File → Preferences (Windows) or Arduino IDE → Settings (macOS).
  • In "Additional Boards Manager URLs", click the icon to add the following link:
https://dl.espressif.com/dl/package_esp32_index.json

(If there are other URLs already, add this one on a new line)

  • Click OK to save.
  • Click the Board Manager icon on the left (or Tools → Board → Board Manager).
  • Search for esp32, find "esp32 by Espressif Systems"
  • Select version 3.0.5 or higher (3.0.x stable version), click Install.
  • After installation, close the Board Manager.

Verification: In Tools → Board → ESP32 Arduino, you should see "ESP32S3 Dev Module" and other options.

1.3 Serial Driver Installation (CH343)

YD-ESP32-S3 development board uses CH343P chip for USB to serial conversion (Type-C port marked COM).

Windows system

Download CH343 driver from WCH official website:

http://www.wch.cn/download/CH343SER_EXE.html
  • Run CH343SER.EXE and click Install.
  • Connect the development board's COM port to the computer with a USB cable.
  • Open Device Manager → Ports (COM and LPT), you should see USB-SERIAL CH343 (COMx).

macOS system

Mostly driver-free, if not recognized, you can install CH34x driver (provided by WCH official website).

Linux system

Kernel ≥5.5 already includes ch341 driver, after connecting, execute ls /dev/ttyUSB* to check.

Common faults

Phenomenon Solution
No new port Change USB cable (must support data transfer); change USB port; restart computer
Yellow exclamation mark (Win) Right-click to update driver → manually select CH343 driver folder
Port exists but cannot upload Close other serial software (such as serial monitor, Cura)

1.4 Board Parameter Configuration (Critical! Must match your hardware version)

YD-ESP32-S3 has two hardware versions, please select the correct Flash and PSRAM configuration according to your module model.

How to identify the version?

Check the silkscreen on the module:

  • ESP32-S3-WROOM-1-N16R8 → 16MB Flash + 8MB PSRAM (corresponds to HW-678A)
  • ESP32-S3-WROOM-1-N8R2 → 8MB Flash + 2MB PSRAM (corresponds to HW-678B)

Configure the following items in the Tools menu of Arduino IDE:

Parameter Recommended value (select based on your hardware) Description
Board ESP32S3 Dev Module General ESP32-S3 development board model
USB CDC On Boot Disabled Use CH343 serial port, do not enable built-in USB serial
CPU Frequency 240MHz (WiFi) Default is fine
Flash Size 16MB (128Mb) or 8MB (64Mb) Match the module, wrong selection will cause upload failure
PSRAM OPI PSRAM (8M for 16MB version) or OPI PSRAM (for N8R2) Note: N8R2 has 2MB PSRAM, need to enable; select OPI PSRAM for N8R2; select Disabled for N4 and other versions without PSRAM
Partition Scheme Default 4MB with spiffs Default is fine for most cases
Upload Speed 921600 Can reduce to 115200 if failed

🔥 Important reminder:

  • If Flash Size is wrong, you will get "Flash config wrong" error during upload.
  • If PSRAM is not enabled correctly, ESP.getPsramSize() will return 0, and later camera and other large memory functions will fail.
  • This development board's RGB LED is connected to GPIO48 (WS2812), which will be used in subsequent experiments.

Verify configuration (can be added in setup()):

Serial.begin(115200);
Serial.printf("Flash size: %d MB\n", ESP.getFlashChipSize() / (1024*1024));
Serial.printf("PSRAM size: %d bytes\n", ESP.getPsramSize());

1.5 First Verification Program: Light up the on-board RGB LED (WS2812)

YD-ESP32-S3 development board has a WS2812 RGB LED onboard, connected to GPIO48. We can use the Adafruit_NeoPixel library to control it and implement simple blinking.

Step 1: Install NeoPixel library

  • Open Arduino IDE, go to Tools → Manage Libraries.
  • Search for Adafruit NeoPixel and install the latest version.

Step 2: Write code

#include 

#define LED_PIN    48
#define LED_COUNT  1

Adafruit_NeoPixel strip(LED_COUNT, LED_PIN, NEO_GRB + NEO_KHZ800);

void setup() {
  strip.begin();
  strip.show();            // Initialize all LEDs to off state
  strip.setBrightness(50); // Set brightness to 50 (0-255)
}

void loop() {
  // Red
  strip.setPixelColor(0, strip.Color(255, 0, 0));
  strip.show();
  delay(1000);
  // Green
  strip.setPixelColor(0, strip.Color(0, 255, 0));
  strip.show();
  delay(1000);
  // Blue
  strip.setPixelColor(0, strip.Color(0, 0, 255));
  strip.show();
  delay(1000);
  // Off
  strip.setPixelColor(0, strip.Color(0, 0, 0));
  strip.show();
  delay(1000);
}

Step 3: Upload program

  • Connect the development board's COM port to the computer with a USB cable.
  • Select the corresponding COM port (Windows) or /dev/cu.usbserial-* (macOS) in Tools → Port.
  • Click the Upload button.
  • Observe if the RGB LED blinks in red → green → blue → off cycle.

Common upload errors and solutions

Error message Reason Solution
Serial port not found Port not selected or driver not installed Re-select port; re-plug USB; install CH343 driver
Timed out waiting for packet header Not in download mode Hold BOOT button, press EN button once, release BOOT, then click upload
Flash config wrong Flash Size wrong Check Flash size (16MB or 8MB)
PSRAM config error PSRAM setting wrong Correctly select OPI PSRAM or Disabled according to module model

Success sign

After upload, the RGB LED starts to cycle through red, green, blue.

If using the factory program, the RGB will stay on after WiFi connection (but this program has overwritten the factory program).

2.3 Pin Usage Taboos (Very Important!)

1. Voltage Limitation: All GPIOs are 3.3V logic level

  • Never directly input 5V voltage, otherwise the chip will be damaged.
  • When connecting 5V modules, must use level conversion circuits (such as TXS0108E) or resistor voltage dividers.
  • For some 5V modules (such as relays), signal input can be connected in series with 1kΩ resistor for current limiting, but long-term reliability is not guaranteed.

2. Special Requirements for Bootstrap Pins

  • GPIO0: Cannot be low level during power-up (unless entering download mode). Internal pull-up during normal use, external buttons are fine.
  • GPIO45/46: Recommended to keep high or floating (internal pull-up) during power-up, do not connect pull-down resistors or buttons.

3. Usage Limitations of USB Pins (GPIO19/20)

  • If USB-OTG function is not needed, these two pins can be used as normal GPIO (3.3V logic).
  • Once USB is initialized (e.g., calling Serial's USB CDC mode or USB.begin()), they are occupied by the USB controller and can no longer be used as GPIO.

4. Dedicated Serial Pins (GPIO43/44)

  • These two pins are already connected to the CH343P chip for UART0 communication.
  • Do not use them as normal I/O, otherwise serial communication will fail, making it impossible to upload programs or print logs.

5. Special Driving for RGB LED (GPIO48)

  • WS2812 uses single-bus serial protocol, cannot be directly controlled with digitalWrite.
  • Must use NeoPixel library or ESP32's RMT peripheral driver.
  • Example code see Chapter 1.

6. Other Notes

  • Some GPIOs will have brief high or low level output during chip startup (e.g., GPIO2 will change during WiFi initialization), try to avoid using them for critical control.
  • Try to avoid using GPIO11~20 as ADC inputs while enabling WiFi (ADC2 conflict).

2.4 Hardware Safety Instructions

Electrostatic Protection (ESD)

  • Touch a metal object (such as computer case) to discharge before touching the development board.
  • It is recommended to use anti-static wristband in dry environment.

Short Circuit Protection

  • Do not touch between pins with metal objects when powered on.
  • Check for solder bridges before soldering headers.
  • Pay attention to correct polarity when connecting external power supply (reverse connection will burn the voltage regulator chip).

Power Overload Protection

  • On-board 3.3V regulator maximum output current is about 500mA (varies by manufacturer).
  • When connecting high-current modules (servos, motors, multiple sensors), use external 3.3V/5V power supply, do not draw power from on-board 3.3V.
  • Each GPIO output current should not exceed 40mA (recommended ≤20mA), driving LED must be connected in series with current limiting resistor (220Ω~1kΩ) — but WS2812 has built-in current limiting, can be directly connected.

Safety Checklist:

  • ✅ Use well-isolated USB data cable.
  • ✅ Measure power and GND for short circuit with multimeter before first power-up.
  • ✅ Connect GND first when hot-plugging.
  • ❌ Do not use bare board on damp or metal surfaces.
  • ❌ Do not touch running chips with hands (may burn).

2.5 Experiment: Verify On-board RGB LED and BOOT Button

To consolidate the knowledge in this chapter, write a simple program: when the BOOT button is pressed, the RGB LED displays red; when released, it displays green.

Wiring: No external wiring required (on-board LED and button are already connected).

Code:

#include 

#define LED_PIN    48
#define BUTTON_PIN 0

Adafruit_NeoPixel strip(1, LED_PIN, NEO_GRB + NEO_KHZ800);

void setup() {
  strip.begin();
  strip.show();
  pinMode(BUTTON_PIN, INPUT_PULLUP);
}

void loop() {
  if (digitalRead(BUTTON_PIN) == LOW) {  // Pressed
    strip.setPixelColor(0, strip.Color(255, 0, 0)); // Red
  } else {
    strip.setPixelColor(0, strip.Color(0, 255, 0)); // Green
  }
  strip.show();
  delay(50); // Simple debounce
}

Effect:

  • When BOOT is not pressed, RGB LED displays green.
  • When BOOT is pressed, displays red.
  • Returns to green after release.

3.1 Basic Structure of Arduino Programs

Every Arduino program (Sketch) must contain two core functions:

void setup() {
  // Initialization code, executed only once at program startup
}

void loop() {
  // Main loop code, executed repeatedly
}

setup() Function

Automatically runs once when the board is powered on or reset.

Usually used for: setting pin modes (pinMode), initializing serial port (Serial.begin), configuring peripherals, etc.

void setup() {
  pinMode(48, OUTPUT);       // Set GPIO48 as output (on-board RGB LED data line, but note WS2812 requires dedicated library)
  Serial.begin(115200);      // Initialize serial port
}

Note: For the on-board RGB LED (WS2812) of YD-ESP32-S3, it cannot be directly controlled with digitalWrite, must use NeoPixel library. However, basic digital output experiments in this chapter can still use external LEDs or idle GPIOs.

loop() Function

Executes repeatedly after setup() completes.

Usually used for: reading sensors, controlling outputs, communication processing, and other main logic.

void loop() {
  digitalWrite(48, HIGH);
  delay(1000);
  digitalWrite(48, LOW);
  delay(1000);
}

Comments

Single-line comment: // to end of line.

Multi-line comment: /* ... */.

3.3 Core Basic Functions of This Board

The YD-ESP32-S3 development board is based on the ESP32-S3 chip, with basically the same functions as standard Arduino, but some functions have extensions.

Pin Mode Setting: pinMode(pin, mode)
pinMode(48, OUTPUT);          // Digital output
pinMode(0, INPUT);            // High impedance input (floating)
pinMode(0, INPUT_PULLUP);     // Internal pull-up input (default high)
pinMode(1, INPUT_PULLDOWN);   // Internal pull-down input (default low)

Internal pull-up/pull-down resistors are approximately 40kΩ~50kΩ.

Digital Output: digitalWrite(pin, value)
digitalWrite(48, HIGH);   // Output 3.3V
digitalWrite(48, LOW);    // Output 0V

⚠️ Note: GPIO48 is connected to WS2812 RGB LED, cannot be directly controlled with digitalWrite. To light up the on-board LED, use NeoPixel library (see Chapter 1). For digital output experiments in this section, it is recommended to use external LEDs or idle GPIOs (such as GPIO1).

Digital Input: digitalRead(pin)
int state = digitalRead(0);   // Read BOOT button state (LOW when pressed)
if (state == LOW) {
  // Button is pressed
}
PWM Output: analogWrite(pin, value)

All GPIOs of ESP32 support PWM (via LEDC peripheral).

value range: 0 (completely off) ~ 255 (completely on).

analogWrite(1, 128);   // GPIO1 outputs 50% duty cycle square wave

For higher precision or frequency adjustment, use ledcSetup() and ledcAttachPin().

ADC Input: analogRead(pin)

12-bit resolution (0~4095), default range 0~3.3V.

ADC1 pins (GPIO1~GPIO10) can be used simultaneously with WiFi.

ADC2 pins (GPIO11~GPIO20) are unavailable when WiFi is on.

int adcValue = analogRead(1);   // Read analog voltage on GPIO1
float voltage = adcValue * (3.3 / 4095.0);
Serial.println(voltage);

Range can be extended to ~3.9V via analogSetAttenuation(ADC_11db).

5.2 Limitations of Blocking Delay (Cannot Execute Multiple Tasks Simultaneously)

Problem Demonstration: Trying to Make LED Blink and Serial Print Once Per Second Simultaneously
#define LED_PIN 1

void setup() {
  pinMode(LED_PIN, OUTPUT);
  Serial.begin(115200);
}

void loop() {
  // LED blinks once (0.5 second cycle)
  digitalWrite(LED_PIN, HIGH);
  delay(250);
  digitalWrite(LED_PIN, LOW);
  delay(250);
  
  // Serial print (expected once per second)
  Serial.println("Hello");
  delay(1000);
}

Actual effect: LED blinking and serial printing are executed sequentially, not simultaneously. Because delay() completely blocks the program, it cannot execute other code during the waiting period.

Fundamental Defects
  • The program is in a "busy waiting" state during delay(), CPU idles but does not respond to other operations.
  • Cannot achieve true simultaneous (concurrent) tasks.
  • For systems that require real-time response (such as reading buttons, processing communication), delay() will cause response delays.

Solution: Clock-based non-blocking delay.

5.3 Non-blocking Delay: millis() Function

Principle

millis() returns the number of milliseconds that have passed since the program started running (board power-on or reset), type unsigned long.

Overflows and resets to zero after about 50 days, but correct handling of differences can avoid problems.

The core of non-blocking delay: record the time of the last action, and continuously check whether the current time has reached the time for the next action.

Non-blocking LED Blinking Example (Replacing delay(500))
#define LED_PIN 1

unsigned long previousMillis = 0;
const long interval = 500;   // Toggle interval (milliseconds)

void setup() {
  pinMode(LED_PIN, OUTPUT);
}

void loop() {
  unsigned long currentMillis = millis();
  
  if (currentMillis - previousMillis >= interval) {
    previousMillis = currentMillis;
    digitalWrite(LED_PIN, !digitalRead(LED_PIN));   // Toggle state
  }
  
  // Other tasks can run non-blockingly
}
Key Points:
  • Use unsigned long to store time.
  • The difference calculation current - previous >= interval can correctly handle overflow.
  • Between condition checks, the program can execute other code.
Multi-task Parallel Implementation: Controlling LED Blinking and Serial Printing Simultaneously
#define LED_PIN 1

unsigned long prevLed = 0;
unsigned long prevSerial = 0;
const long ledInterval = 300;    // LED toggles every 300ms
const long serialInterval = 1000; // Serial prints every 1 second

void setup() {
  pinMode(LED_PIN, OUTPUT);
  Serial.begin(115200);
}

void loop() {
  unsigned long now = millis();
  
  // LED task
  if (now - prevLed >= ledInterval) {
    prevLed = now;
    digitalWrite(LED_PIN, !digitalRead(LED_PIN));
  }
  
  // Serial print task
  if (now - prevSerial >= serialInterval) {
    prevSerial = now;
    Serial.println("Hello from non-blocking loop");
  }
}

Effect: LED blinks with 300ms cycle, while serial port prints once per second, without interfering with each other.

5.4 Microsecond-level Delay: micros()

micros() returns the number of microseconds since the program started (1 second = 1,000,000 microseconds), type unsigned long.

Overflows after about 71 minutes.

Suitable for high-precision short delays (such as sensor pulse measurement).

Example: Generate a 10-microsecond High-level Pulse

  pinMode(pulsePin, OUTPUT);
}

7.1 External Interrupt Principle and Application Scenarios

What are external interrupts?

When the level of a GPIO pin changes (e.g., from high to low), the hardware pauses the currently executing program, jumps to a pre-specified function (called an interrupt service routine, ISR) for execution. After execution, the program returns to its original position and continues running.

Why do we need interrupts?

  • Polling (constantly calling digitalRead in loop) consumes a lot of CPU time, and response delay is uncertain (may miss short pulses).
  • Interrupts can achieve microsecond-level hardware response, very suitable for:
    • Emergency stop buttons.
    • Encoder pulse counting.
    • Sensor signal capture (e.g., echo from ultrasonic modules).
    • Waking up the chip from deep sleep.

ESP32-S3 interrupt features:

  • All GPIO pins can be used as external interrupt sources.
  • Supports multiple trigger modes (see section 7.3).
  • Interrupt priority can be set (default is medium priority).

7.2 Available Interrupt Pins (All GPIOs Support External Interrupts)

All digital GPIOs on the YD-ESP32-S3 development board can be configured for external interrupts, including:

GPIO0~21, GPIO42~48 (avoid occupied special pins like GPIO43/44 used for serial port, but they can also be used as interrupts, it just affects serial communication).

Recommended interrupt pins:

  • On-board BOOT button (GPIO0) - convenient for testing.
  • External buttons or sensors can be connected to free pins like GPIO1, GPIO2.
  • Avoid using GPIO19/20 (dedicated to USB-OTG, can be used if USB is not needed).

7.3 Interrupt Trigger Modes

Set the trigger mode through the third parameter of attachInterrupt(pin, ISR, mode):

Mode Meaning Example Application
RISING Rising edge (low level → high level) Button release (with pull-up resistor)
FALLING Falling edge (high level → low level) Button press (commonly used)
CHANGE Level change (rising or falling edge) Encoder signals
LOW Low level trigger (continuous trigger while level remains low) Emergency stop (⚠️ Use with caution)
HIGH High level trigger (similar to LOW) Rarely used

⚠️ LOW and HIGH modes need to be used cautiously on ESP32, as the interrupt will repeatedly trigger as long as the level is maintained, which may cause the program to freeze.

Common examples:

attachInterrupt(0, buttonISR, FALLING);   // Trigger when button is pressed
attachInterrupt(1, encoderISR, CHANGE);   // Trigger when encoder rotates

7.4 Basic Experiment: Button Triggered Interrupt to Toggle LED

Experiment purpose: When the BOOT button (GPIO0) is pressed, immediately toggle the state of the external LED (GPIO1) through interrupt, without using polling in the loop.

Hardware connection:

  • External LED connected to GPIO1 (in series with 220Ω resistor).
  • On-board BOOT button (GPIO0, already pull-up, low level when pressed).

Complete code:

#define BUTTON_PIN 0
#define LED_PIN 1

// volatile keyword tells the compiler that this variable may be modified in interrupts, disable optimization
volatile bool ledState = false;

void IRAM_ATTR buttonISR() {
  // Cannot use delay(), Serial.print(), etc. in interrupt service routines
  ledState = !ledState;          // Toggle state
  digitalWrite(LED_PIN, ledState);   // Update LED immediately
}

void setup() {
  pinMode(LED_PIN, OUTPUT);
  pinMode(BUTTON_PIN, INPUT_PULLUP);
  digitalWrite(LED_PIN, ledState);
  
  // Configure interrupt: falling edge trigger (when pressed)
  attachInterrupt(digitalPinToInterrupt(BUTTON_PIN), buttonISR, FALLING);
}

void loop() {
  // Main loop can execute other tasks, such as serial printing
  // Note: Do not modify ledState in loop to avoid conflict with interrupts (need to disable interrupts if necessary)
}

Effect: Each time the BOOT button is pressed, the LED state immediately toggles, even if there is a delay(5000) in the main loop, it can respond immediately.

Important function explanations:

  • digitalPinToInterrupt(pin): Converts pin number to interrupt number (ESP32 can directly pass pin number, but for cross-platform compatibility, it is recommended to use this function).
  • IRAM_ATTR: Puts the interrupt service routine into IRAM (internal RAM), ensuring execution even if Flash is occupied. Must be added.
  • volatile: See below.

7.5 The Role of the volatile Keyword

Problem: During code optimization, the compiler may assume that the value of a variable does not change between two accesses, thus caching it in a register. If the variable is modified in an interrupt, the main loop may read the old value (due to optimization).

Solution: Use volatile to modify variables, telling the compiler: "This variable may be modified unexpectedly at any time, and it must be re-read from memory every time it is used".

Correct usage:

volatile bool flag = false;
volatile int counter = 0;
volatile unsigned long timestamp = 0;

Error example (without volatile, may cause strange bugs):

bool flag = false;   // No volatile
// In interrupt: flag = true;
// In main loop: while (!flag);  // May never exit

✅ Rule: All variables shared between interrupt service routines and the main loop must be of volatile type.

7.6 Interrupt Usage Instructions (Very Important!)

1. Functions that cannot be used in interrupt service routines
  • Blocking functions: delay(), delayMicroseconds() (long delays).
  • Serial output: Serial.print(), etc. (may deadlock or take too long).
  • Dynamic memory allocation: new, malloc, String operations.
  • Certain library functions: such as Wire, SPI read/write (usually include delays or locks).
  • millis() / micros(): Can be called, but note that they themselves rely on interrupts, and calling them in interrupts may be inaccurate (not recommended).

Why? During interrupts, other interrupts may be temporarily disabled. If the ISR takes too long, it will delay other interrupt responses and even cause watchdog reset.

2. Interrupt service routines should be as short as possible
  • Only do the most necessary things: such as setting flags, incrementing counters, digital writing.
  • Complex processing should be placed in the main loop: set flags in ISR, and the main loop checks flags and executes time-consuming operations.

Example: Move LED toggle operation to the main loop (avoid directly calling digitalWrite in ISR, although simple operations are allowed, but setting flags is a safer way).

volatile bool buttonPressed = false;

void IRAM_ATTR buttonISR() {
  buttonPressed = true;   // Only set flag
}

void loop() {
  if (buttonPressed) {
    buttonPressed = false;
    digitalWrite(LED_PIN, !digitalRead(LED_PIN));
  }
}
3. Synchronization issues for shared variables

If the main loop and interrupt access the same variable (such as counter++) at the same time, data races may occur. For simple types (such as int), it is generally an atomic operation on 32-bit ESP32, but a safer method is:

Temporarily disable interrupts before accessing shared variables: noInterrupts() / interrupts().

noInterrupts();
int localCopy = counter;
interrupts();
4. Separate interrupt sources

Multiple interrupts can be used simultaneously, but ensure their priority and response time are reasonable. By default, all external interrupts have the same priority.

Hardware: EC11 rotary encoder (CLK connected to GPIO2, DT connected to GPIO3, common terminal connected to GND).

8.1 Principles and Functions of Serial Communication

What is UART serial communication?

UART (Universal Asynchronous Receiver/Transmitter) is a communication protocol that sends data bit by bit. It only requires two signal lines: TX (transmit) and RX (receive). Both communication parties must agree on the same baud rate (such as 9600, 115200). The data format is usually: start bit + 8 data bits + stop bit (optional parity bit).

Role in Arduino development:

  • Debug output: Use Serial.print() to send variable values and status information to the computer.
  • Command control: The computer sends commands through the serial port to control the development board's actions.
  • Communication with external serial devices (such as GPS modules, Bluetooth modules).

YD-ESP32-S3 serial port features:

  • Up to 3 hardware UART controllers (UART0, UART1, UART2).
  • UART0's TX/RX is fixedly connected to the on-board CH343P chip, corresponding to GPIO43 (TX) and GPIO44 (RX).
  • UART1 and UART2 can be mapped to any GPIO.

8.2 Serial Port Description of This Board

UART0: CH343P USB to serial port (default debugging serial port)

TX: GPIO43 (connected to CH343P's RX)

RX: GPIO44 (connected to CH343P's TX)

This serial port is connected to the computer through the Type-C port marked COM, used for program uploading and Serial object communication.

Note: GPIO43/44 are already fixedly connected to CH343P on the board, do not use them as ordinary I/O, otherwise it will affect serial port functionality.

UART1 (mappable external serial port)

No fixed default pins, users can arbitrarily specify TX/RX pins in code.

Example: Serial1.begin(9600, SERIAL_8N1, rxPin, txPin).

Commonly used to connect external serial devices (such as GPS, sensors).

UART2 (optional)

Usage is the same as UART1, operated through the Serial2 object.

Multiple serial port objects:

Object Name Corresponding Hardware Default Pins Remappable

9.1 Timer Principles and Advantages

A hardware timer is a hardware module independent of the main loop that can automatically trigger interrupts at preset time intervals to execute specified tasks.

Advantages:

  • Does not occupy main loop time, the main loop can execute other tasks.
  • High timing precision, not affected by main loop execution time.
  • Can run multiple timers simultaneously to implement tasks with different time intervals.

9.2 Introduction to ESP32-S3 Hardware Timers

ESP32-S3 has 4 built-in general-purpose hardware timers (Timer 0-3), each with a 64-bit counter.

9.3 Basic Experiment: Timer Toggles LED Every Second

Goal: Use hardware timer to toggle LED state every second.

Code:

#include "soc/timer_group_struct.h"
#include "soc/timer_group_reg.h"

const int LED_PIN = 48;
bool ledState = LOW;

// Timer interrupt service function
void IRAM_ATTR onTimer() {
  // Toggle LED state
  ledState = !ledState;
  digitalWrite(LED_PIN, ledState);
  
  // Clear interrupt flag
  TIMERG0.int_clr_timers.t0 = 1;
  // Re-enable timer
  TIMERG0.hw_timer[0].config.alarm_en = TIMER_ALARM_EN;
}

void setup() {
  pinMode(LED_PIN, OUTPUT);
  Serial.begin(115200);
  Serial.println("Hardware timer example started");
  
  // Configure timer
  timer_config_t config;
  config.divider = 80;  // Clock division, 80MHz / 80 = 1MHz
  config.counter_dir = TIMER_COUNT_UP;
  config.counter_en = TIMER_PAUSE;
  config.alarm_en = TIMER_ALARM_EN;
  config.intr_type = TIMER_INTR_LEVEL;
  config.auto_reload = true;
  
  // Initialize timer 0
  timer_init(TIMER_GROUP_0, TIMER_0, &config);
  
  // Set counter value to 0
  timer_set_counter_value(TIMER_GROUP_0, TIMER_0, 0);
  
  // Set alarm value, 1 second = 1,000,000 microseconds
  timer_set_alarm_value(TIMER_GROUP_0, TIMER_0, 1000000);
  
  // Register interrupt callback function
  timer_isr_register(TIMER_GROUP_0, TIMER_0, onTimer, NULL, ESP_INTR_FLAG_IRAM, NULL);
  
  // Start timer
  timer_start(TIMER_GROUP_0, TIMER_0);
}

void loop() {
  // Main loop can execute other tasks
  Serial.print(".");
  delay(500);
}

Timer time calculation method:

  • ESP32 system clock is 80MHz.
  • Set the division through the divider parameter, for example, when divider=80, the timer clock is 1MHz.
  • The unit of alarm value is microseconds, for example, 1000000 represents 1 second.

10.1 PWM Principles and Application Scenarios

PWM (Pulse Width Modulation) is a technology that controls analog quantities by changing the pulse width.

Principles:

  • By quickly switching the output signal, change the duty cycle of the high level within a period to control the analog quantity.
  • Duty cycle = high level time / cycle time.
  • For LEDs, the larger the duty cycle, the higher the brightness.

Application scenarios:

  • LED dimming
  • Motor speed control
  • Servo angle control
  • Audio output

13.1 SPI Principles and Characteristics

SPI is a synchronous serial communication protocol consisting of four lines:

  • SCLK: Serial clock (generated by the master).
  • MOSI: Master output, slave input.
  • MISO: Master input, slave output.
  • SS/CS: Chip select signal (active low).

Characteristics:

  • Full-duplex communication.
  • High-speed transmission (up to tens of MHz).
  • Master-slave architecture, one master can connect multiple slaves.
  • Suitable for short-distance communication.

13.2 SPI Pins of This Board

Default FSPI pins:

  • SCLK: GPIO12
  • MOSI: GPIO11
  • MISO: GPIO13
  • CS: Customizable (such as GPIO10)

3.15 Chapter 15: Bluetooth Communication

15.1 ESP32-S3 Bluetooth Function Introduction

ESP32-S3 has a built-in complete Bluetooth 5.0 Low Energy (BLE) subsystem. It supports various core functions, including long-range transmission using Coded PHY and Advertising Extensions, as well as a 2Mbps high-speed physical layer, which can provide higher data transmission rates and throughput.

However, one key point to note is: ESP32-S3 itself does not support classic Bluetooth (BR/EDR). Therefore, it cannot directly use traditional Bluetooth headphones, Bluetooth speakers, etc., nor can it run classic Bluetooth (SPP) serial port programs based on the BluetoothSerial library.

So, how to implement serial communication between a mobile phone and the development board on ESP32-S3? The answer is: use BLE UART (Bluetooth Low Energy Universal Asynchronous Receiver/Transmitter). This is a mechanism that simulates serial communication through the BLE protocol, allowing ESP32-S3 to transmit data with "Bluetooth Serial Assistant" applications on mobile phones just like classic Bluetooth. The ESP-IDF official also provides relevant demonstration programs to implement this function.

15.1.1 Bluetooth Protocol Stack Selection

In the Arduino environment, there are mainly the following two options for developing BLE applications for ESP32-S3:

  • BluetoothSerial library (not recommended): This library is simple but actually designed for ESP32 chips that support classic Bluetooth. Using it on ESP32-S3 often results in compilation errors (for example, functions like esp_spp_connect, esp_bt_gap_resolve_eir_data are undefined), so it is no longer recommended.
  • ArduinoBLE library (official recommendation): This is an official library specifically for BLE development, powerful and well-supported for ESP32-S3. This is also the method used in the subsequent experiments in this chapter.
  • NimBLE library (advanced option): Compared to the official Bluedroid protocol stack, NimBLE is a more lightweight BLE protocol stack with lower resource consumption. It also supports ESP32-S3 and is compatible with some advanced Bluetooth 5.0 features. If you have higher requirements for performance or memory usage, you can study this library on your own.

15.2.3 Complete Code

The following code shows how to create a BLE UART service. After the mobile phone connects, ESP32-S3 will send a welcome message via Bluetooth, and can print the data sent by the mobile phone to the serial monitor and automatically reply with a confirmation message.

#include <ArduinoBLE.h>

// Define UUIDs for custom service and characteristics
// Use standard Nordic UART Service (NUS) UUID for compatibility with common phone apps
#define BLE_UART_SERVICE_UUID           "6E400001-B5A3-F393-E0A9-E50E24DCCA9E"
#define BLE_UART_RX_CHARACTERISTIC_UUID "6E400002-B5A3-F393-E0A9-E50E24DCCA9E"
#define BLE_UART_TX_CHARACTERISTIC_UUID "6E400003-B5A3-F393-E0A9-E50E24DCCA9E"

BLEService uartService(BLE_UART_SERVICE_UUID);
// RX characteristic (phone -> ESP32-S3)
BLECharacteristic rxCharacteristic(BLE_UART_RX_CHARACTERISTIC_UUID, BLEWrite, 20);
// TX characteristic (ESP32-S3 -> phone)
BLECharacteristic txCharacteristic(BLE_UART_TX_CHARACTERISTIC_UUID, BLENotify, 20);

String receivedData = "";
bool dataReady = false;

void setup() {
  Serial.begin(115200);
  while (!Serial);

  // Initialize BLE module
  if (!BLE.begin()) {
    Serial.println("BLE initialization failed!");
    while (1);
  }

  // Set device name
  BLE.setDeviceName("ESP32S3_BLE_UART");
  BLE.setLocalName("ESP32S3_BLE_UART");

  // Add service and characteristics
  BLE.setAdvertisedService(uartService);
  uartService.addCharacteristic(rxCharacteristic);
  uartService.addCharacteristic(txCharacteristic);
  BLE.addService(uartService);

  // Set characteristic callback function
  rxCharacteristic.setEventHandler(BLEWritten, onDataReceived);

  // Start advertising
  BLE.advertise();

  Serial.println("BLE UART device started, waiting for connection...");
  Serial.print("Device name: ");
  Serial.println(BLE.deviceName());
}

void loop() {
  // Poll BLE central device (e.g., phone) connection status and events
  BLE.poll();
  
  // Handle serial input (ESP32-S3 -> phone)
  if (Serial.available()) {
    String serialInput = Serial.readString();
    // Send data to phone via TX characteristic
    txCharacteristic.writeValue(serialInput.c_str(), serialInput.length());
    Serial.print("Sent via Bluetooth: ");
    Serial.println(serialInput);
  }
  
  // Handle data from phone
  if (dataReady) {
    Serial.print("Received phone data: ");
    Serial.println(receivedData);
    
    // Send confirmation message back to phone
    String ackMsg = "Received: " + receivedData;
    txCharacteristic.writeValue(ackMsg.c_str(), ackMsg.length());
    
    receivedData = "";
    dataReady = false;
  }

  delay(50);
}

// Callback function: Called when phone writes data via RX characteristic
void onDataReceived(BLEDevice central, BLECharacteristic characteristic) {
  String value = characteristic.value();
  if (value.length() > 0) {
    receivedData = value;
    dataReady = true;
  }
}

15.3 Introduction to BLE (Bluetooth Low Energy)

BLE is a Bluetooth standard designed for low power consumption and small data transmission. Compared to traditional Bluetooth, it has the following significant advantages:

  • Extremely low power consumption: This is the core advantage of BLE. It can maintain extremely low energy consumption when broadcasting or maintaining a connection, which is very suitable for devices powered by button batteries.
  • Fast connection: BLE devices can establish connections in milliseconds, without the need for complex pairing and setup like traditional Bluetooth.
  • Advertising mode: BLE supports "Advertising" mode, where devices can continuously send small data (such as beacon information, sensor readings) without establishing connections with any device, which is very suitable for IoT sensor networks.

15.3.1 GATT (Generic Attribute Profile) — The Core Architecture of BLE

GATT is the application layer protocol for data exchange between BLE devices. It defines how data is organized and transmitted. Understanding GATT architecture is crucial for developing BLE applications. The main concepts are as follows:

  • Service: A service is a collection of related "characteristics" used to complete specific functions. For example, a "Battery Service" will contain a "Battery Level" characteristic. A BLE device can contain multiple services.
  • Characteristic: A characteristic is the smallest data unit in GATT, containing a value and a set of properties (such as read, write, notify). The "RX characteristic" and "TX characteristic" in the above experiment are two characteristics used for sending and receiving data.
  • Descriptor: A descriptor is an attribute that provides additional information for a characteristic. For example, the "Client Characteristic Configuration Descriptor" (CCCD) is used to enable or disable "notify" or "indicate" functions.

15.3.2 BLE Communication Modes

In BLE, data interaction is mainly carried out through the following modes:

  • Read/Write: This is the most basic mode, initiated by the central device (such as a mobile phone), reading characteristic values from the peripheral (such as ESP32-S3) or writing new values to characteristics.
  • Notify/Indicate: This is the core of BLE low power mode. The peripheral can actively "push" data to the central device without a request from the central device. "Notify" is a one-way, unacknowledged transmission; "Indicate" requires acknowledgment of data packets to ensure the central device has received them.

15.3.3 BLE Roles (Central and Peripheral)

In BLE connections, devices play different roles:

  • Central device: Usually a more powerful device (such as a mobile phone, computer), responsible for scanning and connecting to other BLE peripherals. A central device can connect to multiple peripherals simultaneously.
  • Peripheral device: Usually a single-function, low-power device (such as a sensor, bracelet, ESP32-S3 development board), waiting for central device connection by broadcasting its presence. Each peripheral can only connect to one central device at a time.

In the experiment of this chapter, ESP32-S3 plays the "peripheral" role, while your mobile phone plays the "central device" role.
