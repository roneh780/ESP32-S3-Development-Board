5. MicroPython Development Guide for ESP32-S3

1. Board Overview and MicroPython Introduction

1.1 Board Hardware Overview

The development board used in this guide is based on the ESP32-S3-WROOM-1 module as the main controller, which is a powerful IoT development board with rich interfaces. Here's a detailed introduction to its core hardware resources.

1.1.1 Main Controller Module: ESP32-S3-WROOM-1

Chip Model: ESP32-S3 (Xtensa® dual-core 32-bit LX7 processor, up to 240 MHz clock speed)

Available Configurations:

  • N16R8: 16 MB Flash + 8 MB PSRAM
  • N8R8: 8 MB Flash + 8 MB PSRAM
  • N8R2: 8 MB Flash + 2 MB PSRAM

Wireless Features: 2.4 GHz Wi-Fi (802.11 b/g/n) and Bluetooth 5 (LE + Mesh)

Features: Supports AI instruction extensions, vector acceleration, large memory capacity, suitable for complex algorithms and multimedia applications

1.1.2 Onboard Key Resources

Resource Description
RGB-LED WS2812 smart RGB light, control pin is GPIO48, active high, can independently display 256 levels of color
USB to Serial CH343P chip, connected to the Type-C interface marked "COM", used for firmware flashing and log printing
USB-OTG Another Type-C interface, can be used as USB Host/Device after shorting the resistor on the board (or using a 0Ω resistor), supports USB drives, keyboards, etc.
Buttons RST button (reset), BOOT button (enter download mode during flashing)
Power LDO voltage regulator, supports 3.3~5V power supply (input through either Type-C port)
IO Ports About 36 available GPIOs, including I2C, SPI, UART, ADC, touch and other peripheral functions

Notes:

  • Before using USB-OTG functionality, you need to short the "USB" pad on the board (usually a 0Ω resistor or shorting jumper).
  • The two Type-C interfaces have different functions: the one with "COM" silkscreen is for serial communication and firmware download; the other "USB" interface is for OTG functionality (can also be used for download by holding BOOT while powering up).

1.1.3 Hardware Pinout Key Points

Common function pins (based on typical board layout):

  • UART0: GPIO43 (TX0), GPIO44 (RX0) → connected to CH343P, communicates with PC
  • BOOT button: GPIO0 (low level when pressed)
  • RGB-LED (WS2812): GPIO48
  • I2C: typically GPIO1(SDA)/GPIO2(SCL) or GPIO4(SDA)/GPIO5(SCL) (refer to schematic for specifics)
  • ADC1: multiple channels including GPIO1~GPIO10

Pin voltage: All are 3.3V logic, do not directly input 5V, otherwise the chip may be damaged

For detailed pin mapping, please refer to the official ESP32-S3 datasheet and development board schematic (available from customer service).

1.2 MicroPython Introduction and Advantages

MicroPython is a streamlined and efficient implementation of Python 3, specifically designed for microcontrollers and embedded systems. It preserves the core syntax of Python while providing direct hardware manipulation interfaces, allowing developers to easily control GPIO, I2C, SPI, WiFi and other peripherals with Python.

1.2.1 Why Choose MicroPython?

  • Easy to learn and use: Python syntax is concise, no complex compilation environment configuration required, suitable for rapid prototyping.
  • Interactive programming: Through serial REPL (interactive interpreter), you can execute code in real-time and debug hardware, similar to "command line operation".
  • Rich library support: Built-in modules like machine, network, neopixel, socket, covering common hardware and network functions.
  • Cross-platform: The same code can run on multiple development boards like ESP32, ESP8266, RP2040 with minor modifications.
  • Strong community: Plenty of open-source examples and third-party libraries, shortening development cycle.

1.2.2 MicroPython Adaptation to ESP32-S3

ESP32-S3 has large PSRAM (up to 8MB) and dual-core processor, and MicroPython firmware has been fully optimized for it:

  • Supports dual-core mode, can implement multi-threading through _thread module.
  • Supports USB-OTG (TinyUSB library), can simulate keyboard, mouse or read/write USB drives.
  • Supports simultaneous WiFi and Bluetooth operation.
  • Built-in NeoPixel (WS2812) driver for direct RGB-LED control.

1.3 Target Audience and Learning Objectives

1.3.1 Target Audience

  • Enthusiasts with some Python foundation who want to get started with IoT hardware
  • Students or engineers in electronics and computer-related majors
  • Designers who need to quickly validate IoT product prototypes
  • Learners who want to switch from Arduino to a more efficient language

1.3.2 Learning Objectives

After completing this guide, you will be able to:

  • Independently set up the ESP32-S3 MicroPython development environment (firmware flashing, IDE configuration).
  • Control onboard resources: light up RGB-LED, read buttons, use serial port for debugging information.
  • Connect external sensors: read temperature and humidity, display OLED through I2C/SPI.
  • Implement WiFi communication: connect to network, send HTTP requests, build simple web server.
  • Apply USB-OTG: read/write USB drives, connect keyboard/mouse.
  • Complete practical projects: such as WiFi clock, remote control RGB light, MQTT data reporting, etc.

2. Development Environment Setup

After completing the hardware introduction in Chapter 1, this chapter will guide you through setting up the ESP32-S3 MicroPython development environment from scratch. The entire process is divided into four steps: firmware acquisition → flashing tool installation → firmware flashing → IDE configuration and testing. This chapter adopts a progressive structure of "toolchain configuration → firmware flashing → IDE debugging", and it is recommended to follow the order.

2.1 Firmware Acquisition

2.1.1 MicroPython Official Firmware Download

MicroPython officially provides precompiled firmware for ESP32-S3, which can be obtained by visiting the official download page.

Open your browser and visit the MicroPython official download page: https://micropython.org/download/ESP32_GENERIC_S3/

You can see multiple versions of firmware files in the list. It is recommended to choose the version with SPIRAM_OCT identifier (corresponding to the PSRAM equipped on your development board). The common configuration of this development board is N16R8 (16MB Flash + 8MB PSRAM), so it is recommended to select the firmware in the format of ESP32_GENERIC_S3-SPIRAM_OCT-xxx.bin.

Download the .bin format firmware file and save it to an easily accessible local path (such as desktop or a dedicated folder). The official website download speed may be slow, and you can also use the firmware file provided in the supporting materials package.

Note: Do not download .dfu or other format files. ESP32-S3 requires .bin format firmware.

2.2 Flashing Tool Installation

To flash MicroPython firmware into the ESP32-S3 Flash, you need to use the esptool.py flashing tool provided by Espressif. This tool is based on Python and can be easily installed through pip.

2.2.1 Install Python (if not already installed)

Visit the Python official website: https://www.python.org/downloads/

Download Python 3.7 or higher version (recommended 3.10+), and make sure to check the "Add Python to PATH" option during installation.

After installation is complete, open the command line (CMD or PowerShell) and enter the following command to verify the installation:

python --version

If the Python version number is displayed, the installation is successful.

2.2.2 Install esptool

Execute the following command in the command line:

pip install esptool

If you encounter permission issues (e.g., on Linux/macOS), you can add the --user parameter:

pip install esptool --user

After installation, you can verify the version:

esptool.py version

Note: On some systems, you may need to use python -m esptool instead of directly calling the esptool command.

2.2.3 Install Serial Driver (CH343P)

This development board uses CH343P as the USB-to-serial chip, and you need to install the corresponding driver to recognize the serial port on the computer.

Windows system:

  • Visit WCH's official download page: https://www.wch.cn/downloads/CH343SER_EXE.html
  • Download the "CH343SER_EXE" one-click installation program.
  • Double-click to run, click the "Install" button, and wait for the installation to complete.

macOS system:

  • Visit WCH's official website to download the CH343 macOS driver package.
  • After decompression, double-click the .dmg file and drag the driver into the "Applications" folder to complete the installation.

Linux system:

The Linux kernel has built-in CH34x series drivers, so no additional installation is usually required. The system will automatically recognize the development board when it is plugged in.

After the driver is installed, connect the development board to the computer via a Type-C cable (please use a USB cable that supports data transmission, not just a charging cable), and open "Device Manager" (Windows) to check if a new COM port (such as COM3, COM4, etc.) appears under "Ports (COM and LPT)". If you can see it, the driver installation is successful and the development board has been correctly recognized.

2.3 Firmware Flashing

2.3.1 Comparison of Two Flashing Methods

Two flashing methods are provided for different user habits:

Comparison Method 1: Command-line flashing (esptool) Method 2: One-click flashing in Thonny
Target users Developers familiar with command line Beginners, those who want one-stop configuration
Flexibility High (customizable erase and flash parameters) Medium (automated process)
Additional dependencies Requires separate installation of esptool Only requires Thonny installation
Recommended scenario Firmware upgrade, batch flashing Initial environment setup

Both methods can complete firmware flashing. It is recommended for beginners to use Method 2 (one-click flashing in Thonny) first, as it is simpler and more intuitive. The following introduces them respectively.

Method 1: Command-line flashing (esptool)

Step 1: Put the development board into download mode

Before flashing MicroPython firmware, you need to put ESP32-S3 into download (flashing) mode:

  • Connect the development board (Type-C interface marked COM) to the computer using a USB data cable.
  • Hold down the BOOT button.
  • Press the RST button once (then release the RST button).
  • Release the BOOT button.

The development board is now in flashing mode, waiting to receive firmware.

Note: For some development boards during first flashing or when firmware is damaged, the above operation is necessary; if the development board already has MicroPython and only needs firmware upgrade, sometimes you can flash directly without pressing the BOOT button, but for safety, it is recommended to perform this operation every time.

Step 2: Check the serial port number corresponding to the development board

Execute the following command in the command line to check available serial ports:

python -m serial.tools.list_ports

Or check in Windows Device Manager under "Ports (COM and LPT)", usually displayed as COM3, COM4, etc.

Step 3: Erase Flash (important)

Before flashing new firmware, it is recommended to erase the original data on Flash to avoid conflicts caused by residual data:

esptool.py --chip esp32s3 --port COM4 erase_flash

Replace COM4 with your actual serial port number.

Step 4: Flash firmware

Execute the following command to flash the firmware (note to modify the firmware path and serial port number):

esptool.py --chip esp32s3 --port COM4 --baud 115200 write_flash 0x0 "firmware_file_path.bin"

Example:

esptool.py --chip esp32s3 --port COM4 --baud 115200 write_flash 0x0 "C:\Users\YourName\Desktop\ESP32_GENERIC_S3-SPIRAM_OCT-xxx.bin"

Key note: The Flash start address of ESP32-S3 is 0x0, which is different from ESP32 (address is 0x1000). Be sure not to fill it wrong.

Step 5: Verify flashing success

After flashing is completed, the command line will display success prompts such as "Hash of data verified". At this time, press the RST button to reset the development board and start it, and you can enter the MicroPython environment.

Method 2: One-click flashing in Thonny (recommended for beginners)

Step 1: Install Thonny

  • Visit Thonny official website: https://thonny.org/
  • Download the corresponding installation package according to your operating system.
  • Run the installation program and follow the prompts to complete the installation (it is recommended to use an English path for the installation path, do not include Chinese characters).

Step 2: Connect the development board

Connect the development board to the computer via a USB cable (using the Type-C interface marked COM).

Step 3: Open Thonny and configure the interpreter

  • Open Thonny and click Menu → Run → Configure Interpreter (or directly click the Python version area in the lower right corner of the Thonny window to quickly access).
  • In the pop-up window, select "MicroPython (ESP32)".
  • In the "Port" drop-down box, select the COM port corresponding to the development board.

Step 4: Flash firmware

  • Click the "Install or update MicroPython (esptool)" button in the lower right corner of the window.
  • Click the "≡" icon next to "Install" and select "Select local MicroPython image...".
  • In the pop-up file selection dialog, find and select the .bin firmware file you downloaded earlier, and click "Open".
  • After confirming that "Target port" is the correct COM port, click "Install" to start flashing.
  • Wait for the progress bar to complete, and display "Done!" flashing success prompt.

Step 5: Complete configuration

After flashing is completed, click "OK", and the Shell window at the bottom of Thonny will appear with the >>> prompt, indicating that you have successfully connected to the ESP32-S3 REPL, and the environment setup is complete.

2.5 First MicroPython Program

2.5.1 Run directly through REPL

Enter the following code in the Thonny Shell window:

from machine import Pin
import time

led = Pin(2, Pin.OUT)   # Assume onboard LED is on GPIO2 (adjust according to actual development board)
led.value(1)            # Turn on LED
time.sleep(1)           # Delay 1 second
led.value(0)            # Turn off LED

Note: The onboard LED pins of different development boards may be different. If GPIO2 has no response, you can check the development board schematic to confirm the actual pin number of the onboard LED. The WS2812 RGB-LED control pin of this development board is GPIO48, but WS2812 needs to be controlled through the NeoPixel library, which will be detailed in Chapter 4. This is just to verify the REPL function.

2.5.2 Save and run script

  • Enter the above code in the Thonny code editing area.
  • Click Menu → File → Save As...
  • In the pop-up window, select "MicroPython Device", save the file to the development board, and name the file main.py.
  • Press the RST button to reset the development board, and the program will run automatically.

2.5.3 Testing and debugging tips

  • Press Ctrl+C in the Shell to interrupt the running program.
  • Press Ctrl+D to perform a soft reset (software reset, does not clear memory).
  • Enter help() to view help information for MicroPython built-in modules.
  • Press Ctrl+E to enter paste mode, where you can paste multiple lines of code into the REPL for execution at once.

2.6 Common Issues and Solutions

Q1: Computer cannot recognize the development board serial port (no new port in Device Manager)

Reason 1: CH343P driver is not installed correctly.

Solution: Re-download and install the CH343P driver, then re-plug the USB cable after installation.

Reason 2: The USB cable is only for charging and does not support data transmission.

Solution: Replace with a USB cable that supports data transmission.

Q2: Prompt "Failed to connect to ESP32-S3" during flashing

Reason: The development board is not in download mode.

Solution: Ensure strictly following the sequence of "hold BOOT → press RST → release RST → release BOOT".

Q3: Cannot connect to REPL after flashing, no >>> prompt in Shell window

Reason: Firmware flashing is incomplete or Flash is not erased.

Solution: First execute esptool.py erase_flash to erase the entire Flash, then re-flash the firmware.

Q4: Prompt "write_flash address 0x0" error during flashing

Reason: The Flash start address of ESP32-S3 is 0x0. If you use ESP32's flashing parameters (such as 0x1000), an error will be reported.

Solution: Confirm that the address in the flashing command is 0x0.

Q5: Progress bar gets stuck or no response for a long time when flashing with Thonny

Solution: Disconnect the development board connection, close Thonny, reopen it and follow the steps to re-flash. If it still fails, you can use the command-line esptool method.

3. GPIO Basic Control

This chapter will introduce the basic usage methods of the general-purpose input/output (GPIO) pins of ESP32-S3. You will learn how to configure pins as digital outputs (controlling LED on/off) and digital inputs (reading button status), as well as how to configure internal pull-up/pull-down resistors. 3.2 Digital Output: Controlling LED

Digital output is used to control switch-type devices such as LEDs, relays, buzzers, etc. We will use an external LED (or an LED connected to an unused GPIO on the development board) for demonstration.

3.2.1 Hardware Connection

Prepare a 5mm LED and a 220Ω~1kΩ current-limiting resistor. Connect the LED's positive electrode (long leg) to GPIO4 through the resistor, and the negative electrode to the development board's GND.

If you don't have an external LED temporarily, you can also use the onboard WS2812 RGB-LED for testing, but WS2812 is not a regular LED and requires a dedicated library (which will be detailed in Chapter 4). This chapter recommends using an external LED to learn basic output.

3.2.2 Controlling LED On/Off

In MicroPython, first import the Pin class from the machine module, then create a Pin object and set it to output mode.

from machine import Pin
import time

# Configure GPIO4 as output mode
led = Pin(4, Pin.OUT)

# Turn on LED (high level)
led.value(1)
time.sleep(1)

# Turn off LED (low level)
led.value(0)
time.sleep(1)

# You can also use led.on() and led.off() methods
led.on()   # Turn on
time.sleep(1)
led.off()  # Turn off

3.2.3 Blinking LED (Classic Example)

from machine import Pin
import time

led = Pin(4, Pin.OUT)

while True:
    led.value(1)
    time.sleep(0.5)
    led.value(0)
    time.sleep(0.5)

Press Ctrl+C to terminate the loop.

3.2.4 Using PWM to Control LED Brightness (Preview)

Although GPIO digital output only has high/low two states, different brightness levels can be simulated through Pulse Width Modulation (PWM). PWM will be detailed in Chapter 10, here is a simple breathing light effect preview.

from machine import Pin, PWM
import time

led_pwm = PWM(Pin(4), freq=1000)  # Frequency 1kHz

# Gradually increase duty cycle (0~1023)
while True:
    for duty in range(0, 1024, 10):
        led_pwm.duty(duty)
        time.sleep_ms(10)
    for duty in range(1023, -1, -10):
        led_pwm.duty(duty)
        time.sleep_ms(10)

3.3 Digital Input: Reading Button Status

The development board has a BOOT button connected to GPIO0, and when the button is pressed, the pin becomes low level (GND), and when not pressed, it remains high level due to internal pull-up (or external pull-up). Therefore, we can read the level of GPIO0 to determine whether the button is pressed.

3.3.1 Reading BOOT Button

from machine import Pin
import time

# Configure GPIO0 as input mode (internal pull-up already exists, PULL_UP can be omitted)
button = Pin(0, Pin.IN)

while True:
    if button.value() == 0:   # Low level when pressed
        print("BOOT button pressed!")
        # Avoid repeated triggering, wait for button release
        while button.value() == 0:
            time.sleep_ms(10)
    time.sleep(0.1)

3.3.2 Explicitly Enabling Internal Pull-up Resistor

By default, GPIO0 already has an external pull-up resistor on the development board, but if you use other pins without external pull-up as input, you can enable the chip's internal pull-up resistor to avoid uncertain level when the pin is floating.

# Enable internal pull-up resistor
button = Pin(0, Pin.IN, Pin.PULL_UP)

Similarly, you can use Pin.PULL_DOWN to enable internal pull-down resistor (ESP32-S3 supports internal pull-down).

Note: GPIO0 already has external pull-up, if you enable internal pull-up again, the effect is still high level, no conflict will occur.

3.3.3 Button Debouncing

Mechanical buttons produce multiple level transitions (bouncing) at the moment of pressing and releasing. If not handled, it may cause program misjudgment. A common method for software debouncing is to detect a press and then delay for a short time to confirm the state.

from machine import Pin
import time

button = Pin(0, Pin.IN, Pin.PULL_UP)

def read_button_debounced(pin, delay_ms=50):
    if pin.value() == 0:
        time.sleep_ms(delay_ms)
        if pin.value() == 0:
            # Wait for button release
            while pin.value() == 0:
                time.sleep_ms(10)
            return True
    return False

while True:
    if read_button_debounced(button):

4. Onboard RGB-LED (WS2812) Programming

WS2812 (also known as NeoPixel) is an intelligent RGB-LED with integrated driver circuit, which can achieve 256-level brightness, full-color display and cascade control with just one data line. This development board has an onboard WS2812 RGB-LED connected to GPIO48. This chapter will learn how to use MicroPython's neopixel library to control this RGB-LED, realize single-color display, color mixing, rainbow effects, etc., and make smart indicator lights combined with WiFi connection status.

4.1 WS2812 Introduction

4.1.1 What is WS2812

WS2812 is a highly integrated intelligent LED that integrates RGB-LED chip, constant current driver circuit and single-wire communication protocol controller. This means that only one GPIO pin is needed to control the brightness and color of the LED, and it supports cascading multiple LEDs (the first LED processes its own data and then passes the remaining data to the next LED).

4.1.2 Working Principle

WS2812 uses a unique single-wire return-to-zero (NRZ) communication protocol, encoding data by precisely controlling the duration of high and low levels of IO pins.

  • Each LED requires 24 bits of data (GRB order: 8 bits green + 8 bits red + 8 bits blue)
  • Each color channel has 256 levels of brightness (0~255), which can combine to produce about 16.77 million colors
  • After sending data for all LEDs, the bus needs to be pulled low for more than 50μs (reset pulse) before the LEDs will light up

Data format (24 bits per LED): G7 G6 ... G0 R7 R6 ... R0 B7 B6 ... B0

← 8 bits green → ← 8 bits red → ← 8 bits blue →

WS2812 has high timing accuracy requirements, but MicroPython's built-in neopixel library has handled the timing details, so we don't need to care about the underlying implementation.

4.1.3 Notes

  • Timing sensitivity: WS2812 has strict timing requirements, so when controlling, make sure there are no long blocking operations in the code, such as performing a lot of calculations or long time.sleep during data transmission.
  • Power supply considerations: A single WS2812 has a peak current of about 60mA in RGB full white (255,255,255) state. This development board has only one RGB-LED and can be directly powered by the board's 3.3V. If external multiple WS2812 light strips are connected, an external power supply is required.
  • Voltage matching: 3.3V logic level can generally drive WS2812 directly. However, when connecting long-distance light strips, it is recommended to connect a 33Ω~47Ω resistor in series near the MCU on the data pin to suppress signal reflection.

The rainbow effect is achieved by smoothly traversing the HSV color wheel. The wheel() function maps position values (0~255) to RGB colors:

5. Button Input and Interrupts

Buttons are one of the most commonly used human-computer interaction devices in embedded systems. This chapter will learn how to use MicroPython to read the BOOT button on the development board and implement two button detection methods: polling and external interrupts. At the same time, it will solve the common "bouncing" problem of mechanical buttons, and comprehensively apply the knowledge learned to control the color switching of the onboard RGB-LED through buttons.

5.1 Polling Method to Detect BOOT Button

Polling is the simplest input detection method: the program continuously reads the pin level in a loop and performs corresponding operations when a valid change is detected.

5.1.1 Basic Polling Example

The BOOT button on the development board is connected to GPIO0 and has been pulled up to high level (3.3V) through an external resistor. When the button is pressed, GPIO0 is pulled low (0); when released, it returns to high level (1).

from machine import Pin
import time

button = Pin(0, Pin.IN, Pin.PULL_UP)   # Pull-up mode (external pull-up already exists, can be omitted)

while True:
    if button.value() == 0:            # Pressed
        print("Button pressed")
        # Wait for release to avoid repeated printing
        while button.value() == 0:
            time.sleep_ms(10)
    time.sleep_ms(50)                  # Short delay to reduce CPU usage

5.1.2 Advantages and Disadvantages of Polling

Advantages Disadvantages
Simple to implement, no additional configuration required CPU is always occupied, even when there are no button events
Suitable for scenarios with low response speed requirements If there is a long delay in the loop (such as time.sleep(1)), button response will be delayed
No involvement of interrupt context, easy to debug Cannot work in low-power sleep mode

Improvement direction: You can add time.sleep_ms(10) in the loop to reduce polling frequency, but this will introduce a maximum response delay of 10ms. For most interactive scenarios, this delay is acceptable.

5.2 External Interrupt (IRQ) for Button Response

External interrupts (Interrupt Request) allow the CPU to pause the current task when the pin level changes and immediately execute a predefined callback function. This method has fast response speed, and the CPU can enter sleep or perform other tasks when not processing buttons.

5.2.1 Basic Concepts of Interrupts

Trigger modes：

  • Pin.IRQ_FALLING: Falling edge trigger (high→low, when button is pressed)
  • Pin.IRQ_RISING: Rising edge trigger (low→high, when button is released)
  • Pin.IRQ_LOW_LEVEL: Low level trigger (level sensitive, generally not recommended)
  • Pin.IRQ_HIGH_LEVEL: High level trigger

Callback function: When an interrupt is triggered, the system will call this function. The callback function must be as short as possible, avoiding time-consuming operations (such as delays, printing large amounts of data, memory allocation), otherwise it may affect system stability or cause interrupt nesting.

5.2.2 Using External Interrupts to Detect Button Presses

from machine import Pin
import time

button = Pin(0, Pin.IN, Pin.PULL_UP)

# Interrupt callback function
def button_handler(pin):
    # Note: This function runs in the interrupt context, should be as short as possible
    print("Button interrupt triggered")

# Configure falling edge interrupt (triggered when pressed)
button.irq(trigger=Pin.IRQ_FALLING, handler=button_handler)

# Main program continues to do other things
while True:
    print("Main loop running...")
    time.sleep(2)

Running result: Each time the BOOT button is pressed, the console will print a message, while the main loop continues to print every 2 seconds.

5.2.3 Distinguishing Press and Release

By registering both rising edge and falling edge interrupts and reading the current level in the callback function, you can distinguish between press and release events.

from machine import Pin

button = Pin(0, Pin.IN, Pin.PULL_UP)

def button_handler(pin):
    if pin.value() == 0:
        print("Button pressed")
    else:
        print("Button released")

# Register both falling and rising edges
button.irq(trigger=Pin.IRQ_FALLING | Pin.IRQ_RISING, handler=button_handler)

# Keep program running
while True:
    pass

5.2.4 Interrupt Callback Function Limitations and Best Practices

Due to MicroPython's interrupt handling mechanism, the following functions cannot be used in callback functions：

  • Dynamic memory allocation (such as creating lists, dictionaries, strings, etc.)
  • Using blocking delays such as time.sleep() or time.sleep_ms()
  • Performing operations that may trigger garbage collection
  • Using print() for large amounts of output (small amounts are okay but not recommended)

Recommended practice: Only set a flag variable in the callback function (such as global pressed_flag = True), then check the flag in the main loop and perform time-consuming operations.

from machine import Pin
import time

button = Pin(0, Pin.IN, Pin.PULL_UP)
button_pressed = False

def button_handler(pin):
    global button_pressed
    # 仅设置标志，不执行复杂操作
    button_pressed = True

button.irq(trigger=Pin.IRQ_FALLING, handler=button_handler)

while True:
    if button_pressed:
        print("Button pressed detected in main loop")
        # 执行耗时操作
        button_pressed = False
    # 其他任务...
    time.sleep_ms(100)

5.3 Button Debouncing

Mechanical buttons, at the moment of pressing and releasing, produce multiple level transitions due to elastic collision of metal shrapnel (usually lasting 5~20ms). Without debouncing, a single button press may be mistaken for multiple triggers.

5.3.1 Hardware Debouncing

Connecting a 0.1μF capacitor in parallel across the button can filter out high-frequency bouncing. This development board already includes such hardware debouncing circuit, but for high-reliability scenarios, software-assisted debouncing is still needed.

5.3.2 Software Debouncing

A common method for software debouncing is: after detecting a level change, delay for a short period (such as 20~50ms), then read the level again to confirm if the state is stable.

Implementing debouncing in polling：

from machine import Pin
import time

button = Pin(0, Pin.IN, Pin.PULL_UP)

def read_button_debounced():
    if button.value() == 0:                # Detect press
        time.sleep_ms(30)                  # Delay for debouncing
        if button.value() == 0:            # Confirm again
            # Wait for button release
            while button.value() == 0:
                time.sleep_ms(10)
            return True
    return False

while True:
    if read_button_debounced():
        print("Button confirmed pressed")
    time.sleep_ms(10)

Implementing debouncing in interrupts：

Since delays cannot be used in interrupt callback functions, the debouncing logic can be moved to the main loop.

from machine import Pin
import time

button = Pin(0, Pin.IN, Pin.PULL_UP)
interrupt_flag = False
last_trigger_time = 0

def button_handler(pin):
    global interrupt_flag
    interrupt_flag = True

button.irq(trigger=Pin.IRQ_FALLING, handler=button_handler)

while True:
    if interrupt_flag:
        interrupt_flag = False
        # Debouncing: Check if time since last trigger is greater than 50ms
        current_time = time.ticks_ms()
        if time.ticks_diff(current_time, last_trigger_time) > 50:
            # Confirm level again (after 20ms wait)
            time.sleep_ms(20)
            if button.value() == 0:
                print("Button pressed (debounced)")
                # Wait for release
                while button.value() == 0:
                    time.sleep_ms(10)
        last_trigger_time = current_time
    time.sleep_ms(10)

5.4 Comprehensive Example: Button-Controlled RGB Color Switching

    (255, 0, 0),    # Red

    (0, 0, 255),    # Blue

    (0, 255, 255),  # Cyan

6. Serial Communication (UART)

Serial communication (UART, Universal Asynchronous Receiver/Transmitter) is one of the most commonly used communication interfaces in embedded systems. This development board provides two ways to perform serial communication with external devices (such as PC, sensors, other microcontrollers): one is through the CH343P USB-to-serial chip connected to the PC's Type-C interface (marked "COM"); the other is through the USB-OTG Type-C interface to simulate a virtual serial port (CDC). This chapter will learn how to use MicroPython to operate the hardware UART of ESP32-S3, realize data sending and receiving, and interact with PC serial port debugging assistant.

6.1 Relationship Between Hardware Serial Port and USB-to-Serial

6.1.1 UART Resources of ESP32-S3

ESP32-S3 has built-in 3 hardware UART controllers (UART0, UART1, UART2), each of which can independently configure parameters such as baud rate, data bits, stop bits, and parity bits.

MicroPython does not enable USB-CDC by default on ESP32-S3, it needs to be manually enabled. There are two ways：

Method 1: Enable in boot.py (recommended)

import usb_cdc

7.2 Driving Common I2C Sensors (e.g., AHT10, SSD1306 OLED)

7.2.1 Driving AHT10 Temperature and Humidity Sensor

AHT10 is a high-precision temperature and humidity sensor that uses I2C communication.

from machine import I2C, Pin
import time

class AHT10:
    def __init__(self, i2c, address=0x38):
        self.i2c = i2c
        self.address = address
        self.init_sensor()
    
    def init_sensor(self):
        # Initialize sensor
        self.i2c.writeto(self.address, b'\x71')
        time.sleep_ms(10)
        # Check status
        status = self.i2c.readfrom(self.address, 1)[0]
        if not (status & 0x08):
            print("AHT10 initialization failed!")
    
    def measure(self):
        # Trigger measurement
        self.i2c.writeto(self.address, b'\xac\x33\x00')
        time.sleep_ms(80)  # Wait for measurement to complete
        
        # Read data
        data = self.i2c.readfrom(self.address, 6)
        
        # Parse data
        humidity = ((data[1] << 12) | (data[2] << 4) | (data[3] >> 4)) * 100 / (1 << 20)
        temperature = (((data[3] & 0x0F) << 16) | (data[4] << 8) | data[5]) * 200 / (1 << 20) - 50
        
        return humidity, temperature

# Initialize I2C
i2c = I2C(0, scl=Pin(1), sda=Pin(2), freq=400000)

# Create AHT10 instance
sensor = AHT10(i2c)

# Read data
while True:
    humidity, temperature = sensor.measure()
    print("Temperature: {:.2f}°C, Humidity: {:.2f}%".format(temperature, humidity))
    time.sleep(2)

7.2.2 Driving SSD1306 OLED Display

SSD1306 is a commonly used 128x64 pixel OLED display that uses I2C communication.

from machine import I2C, Pin
import time

class SSD1306:
    def __init__(self, i2c, address=0x3c, width=128, height=64):
        self.i2c = i2c
        self.address = address
        self.width = width
        self.height = height
        self.buf = bytearray(width * height // 8)
        self.init_display()
    
    def init_display(self):
        # Initialization command sequence
        init_cmds = [
            0xAE,  # Turn off display
            0xD5, 0x80,  # Set display clock
            0xA8, 0x3F,  # Set multiplex rate
            0xD3, 0x00,  # Set display offset
            0x40,  # Set start line
            0x8D, 0x14,  # Enable charge pump
            0x20, 0x00,  # Set memory addressing mode
            0xA1,  # Horizontal flip
            0xC8,  # Vertical flip
            0xDA, 0x12,  # Set COM pins
            0x81, 0xCF,  # Set contrast
            0xD9, 0xF1,  # Set pre-charge period
            0xDB, 0x40,  # Set VCOMH
            0xA4,  # Display all pixels
            0xA6,  # Normal display
            0xAF   # Turn on display
        ]
        for cmd in init_cmds:
            self._send_cmd(cmd)
    
    def _send_cmd(self, cmd):
        self.i2c.writeto(self.address, b'\x00' + bytes([cmd]))
    
    def _send_data(self, data):
        self.i2c.writeto(self.address, b'\x40' + data)
    
    def clear(self):
        self.buf = bytearray(len(self.buf))
        self.show()
    
    def show(self):
        # Set display area
        self._send_cmd(0x21)
        self._send_cmd(0)
        self._send_cmd(self.width - 1)
        self._send_cmd(0x22)
        self._send_cmd(0)
        self._send_cmd(self.height // 8 - 1)
        # Send data
        self._send_data(self.buf)
    
    def pixel(self, x, y, color):
        if x < 0 or x >= self.width or y < 0 or y >= self.height:
            return
        index = x + (y // 8) * self.width
        if color:
            self.buf[index] |= 1 << (y % 8)
        else:
            self.buf[index] &= ~(1 << (y % 8))

# Create SSD1306 instance

# Display a pixel

display.show()
time.sleep(2)

# Clear screen
display.clear()

10.2 Hardware PWM Output: Controlling LED Brightness, Servo

PWM (Pulse Width Modulation) is a technique that controls output power by changing the duty cycle of a signal, which can be used to control LED brightness, servo angles, etc.

10.2.1 Basic Usage

from machine import Pin, PWM
import time

# Initialize PWM (GPIO4, 1kHz frequency)
pwm = PWM(Pin(4), freq=1000, duty=0)

# Gradually increase brightness
for duty in range(0, 1024, 10):
    pwm.duty(duty)
    time.sleep_ms(50)

# Gradually decrease brightness
for duty in range(1023, -1, -10):
    pwm.duty(duty)
    time.sleep_ms(50)

# Turn off PWM
pwm.deinit()

10.2.2 Controlling Servo

Servos typically use a 50Hz (20ms period) PWM signal, with angle controlled by duty cycle:

  • 0.5ms pulse width (2.5% duty cycle): 0°
  • 1.5ms pulse width (7.5% duty cycle): 90°
  • 2.5ms pulse width (12.5% duty cycle): 180°
from machine import Pin, PWM
import time

# Initialize PWM (GPIO5, 50Hz frequency)
servo = PWM(Pin(5), freq=50, duty=0)

# Convert servo angle to duty cycle (0-1023)
def set_servo_angle(angle):
    # Angle range: 0-180
    # Duty cycle range: 2.5%-12.5% corresponds to duty: 26-128
    duty = int((angle / 180) * 102 + 26)
    servo.duty(duty)

# Test different angles
angles = [0, 45, 90, 135, 180, 90, 0]
for angle in angles:
    print(f"Setting angle: {angle}°")
    set_servo_angle(angle)
    time.sleep(1)

# Turn off PWM
servo.deinit()

10.3 RTC and Low-Power Mode Introduction

RTC (Real-Time Clock) can maintain time in low-power mode, and low-power mode can significantly extend battery life.

10.3.1 RTC Usage

from machine import RTC
import time

# Initialize RTC

# Set time (year, month, day, weekday, hour, minute, second, microsecond)
rtc.datetime((2023, 12, 25, 1, 10, 30, 0, 0))

# Read time

# Formatted output
t = rtc.datetime()
print(f"{t[0]}-{t[1]:02d}-{t[2]:02d} {t[4]:02d}:{t[5]:02d}:{t[6]:02d}")

# Wait for a while
print("Waiting 5 seconds...")
time.sleep(5)

# Read time again
print("Time after 5 seconds:", rtc.datetime())

10.3.2 Low-Power Mode

ESP32-S3 supports multiple low-power modes, with deep sleep mode being the most commonly used.

from machine import RTC, Pin, deepsleep
import time

# Initialize RTC
rtc = RTC()

# Set wake-up time (after 10 seconds)
rtc.alarm(RTC.ALARM0, 10000)  # 10000ms = 10s

# Enter deep sleep
print("Going to deep sleep for 10 seconds...")
deepsleep()

# Execution will start from here after waking up
print("Woke up from deep sleep!")
print("Current time:", rtc.datetime())

10.3.3 External Wake-up

In addition to RTC alarm, ESP32-S3 in deep sleep can also be woken up by external pins.

from machine import Pin, deepsleep

# Configure wake-up pin
wake_pin = Pin(0, Pin.IN, Pin.PULL_UP)

# Enter deep sleep, set external wake-up
print("Going to deep sleep, wake up by pressing BOOT button...")
deepsleep(wake_pin.irq(trigger=Pin.IRQ_FALLING))

# Execute after waking up
print("Woke up from deep sleep by external interrupt!")

10.3.4 Notes

  • During deep sleep, all peripherals except RTC and wake-up pins are turned off.
  • Deep sleep will lose data in SRAM, so important data needs to be saved to Flash or using RTC memory.
  • After waking up, the program will start executing from the beginning, not from the position before sleep.

11.4 Project Practice 2: Button-controlled RGB Color + USB Serial Command Control

This project allows users to switch RGB colors using the BOOT button and control via USB serial commands.

11.4.1 Implementation

import neopixel
from machine import Pin, UART
import time

# Initialize RGB-LED
np = neopixel.NeoPixel(Pin(48), 1)

# Initialize button (BOOT button on GPIO0)
button = Pin(0, Pin.IN, Pin.PULL_UP)

# Initialize UART0 for serial communication
uart = UART(0, baudrate=115200, tx=43, rx=44)

# Predefined colors
colors = [
    (255, 0, 0),    # Red
    (0, 255, 0),    # Green
    (0, 0, 255),    # Blue
    (255, 255, 0),  # Yellow
    (0, 255, 255),  # Cyan
    (255, 0, 255),  # Magenta
    (255, 255, 255) # White
]
color_index = 0

# Button debounce variables
last_button_state = 1
last_debounce_time = 0
DEBOUNCE_DELAY = 50

# Set initial color
np[0] = colors[color_index]
np.write()
print("Initial color: Red")
print("Press BOOT button to change color")
print("Send 'RGB:R,G,B' via serial to set custom color")

while True:
    # Check button state with debounce
    current_button_state = button.value()
    current_time = time.ticks_ms()
    
    if current_button_state != last_button_state:
        last_debounce_time = current_time
    
    if time.ticks_diff(current_time, last_debounce_time) > DEBOUNCE_DELAY:
        if current_button_state == 0 and last_button_state == 1:
            # Button pressed
            color_index = (color_index + 1) % len(colors)
            np[0] = colors[color_index]
            np.write()
            print(f"Color changed to: {['Red', 'Green', 'Blue', 'Yellow', 'Cyan', 'Magenta', 'White'][color_index]}")
        last_button_state = current_button_state
    
    # Check serial input
    if uart.any():
        data = uart.readline().decode().strip()
        if data.startswith("RGB:"):
            try:
                # Parse RGB values
                rgb_str = data.split(":")[1]
                r, g, b = map(int, rgb_str.split(","))
                # Clamp values to 0-255
                r = min(255, max(0, r))
                g = min(255, max(0, g))
                b = min(255, max(0, b))
                # Set color
                np[0] = (r, g, b)
                np.write()
                print(f"Custom color set: ({r}, {g}, {b})")
            except:
                print("Invalid RGB format. Use 'RGB:R,G,B'")
    
    time.sleep(0.01)

12. Common Issues and Debugging Tips

This chapter addresses common issues you may encounter when working with ESP32-S3 and provides debugging tips to help you resolve them.

12.1 Pin Short Circuit Protection Reminder

ESP32-S3 pins are not protected against short circuits. Proper precautions should be taken to avoid damaging the chip.

12.1.1 Protection Measures

  • Use current-limiting resistors: Always use resistors (e.g., 220Ω) when driving LEDs or other loads directly from GPIO pins.
  • Check wiring: Double-check connections before powering up the board.
  • Avoid 5V signals: ESP32-S3 uses 3.3V logic. Do not connect 5V signals directly to GPIO pins.
  • Use input protection: For external inputs, consider adding protection diodes or transient voltage suppressors.

12.1.2 What to Do If a Short Circuit Occurs

  1. Immediately disconnect power.
  2. Inspect the board for visible damage (burn marks, etc.).
  3. Check the affected pins with a multimeter for continuity to ground.
  4. If damage is severe, the board may need to be replaced.

12.2 Firmware Update Failure Handling

Firmware update failures can occur due to various reasons. Here's how to troubleshoot them.

12.2.1 Common Causes and Solutions

Issue Possible Cause Solution
Failed to connect to ESP32-S3 Not in download mode Follow the correct procedure: hold BOOT, press RST, release RST, release BOOT

  1. Try erasing the entire flash first: esptool.py --chip esp32s3 --port COM4 erase_flash
  2. Use a lower baud rate for flashing: --baud 921600 or --baud 115200
  3. Try a different USB cable and port
  4. Check if the board is getting sufficient power

12.3 Memory and Performance Optimization Tips

ESP32-S3 has limited memory resources. Here are some tips to optimize your code.

12.3.1 Memory Optimization

  • Use efficient data types: Prefer integers over floats when possible.
  • Avoid string concatenation: Use lists to collect strings and join them at the end.
  • Limit global variables: Use local variables where possible.
  • Free memory when done: Explicitly delete large objects when they're no longer needed.
  • Use gc module: Manually trigger garbage collection with import gc; gc.collect() after memory-intensive operations.

12.3.2 Performance Optimization

  • Avoid blocking operations: Use non-blocking code or timers instead of long delays.
  • Optimize loops: Minimize operations inside loops.
  • Use native modules: For performance-critical code, consider using C modules or inline assembly.
  • Limit network operations: Batch network requests and use appropriate timeouts.
